# SDPE 운영자 콘솔 — 유즈케이스 정의서

**문서 번호**: SDPE-UC-001  
**버전**: v1.0

**분류**: 내부 기술문서

## 1. 문서 개요

### 1.1 목적

본 문서는 SAR Data Processing Element (SDPE) 운영자 콘솔의 유즈케이스를 정의한다. 운영자 콘솔은 SAR 위성 데이터 처리 파이프라인의 구성·실행·모니터링·장애 대응을 위한 사내 전용 관리 도구이며, 본 문서는 시스템이 제공해야 하는 기능 범위를 사용자 관점에서 기술한다.

### 1.2 적용 범위

본 문서는 SDPE 운영자 콘솔의 모든 사용자 기능을 대상으로 한다. 외부 사용자 서비스(LIID) 및 OGC 지도 웹서비스 인터페이스는 본 문서의 범위에 포함하지 않는다.

### 1.3 관련 문서

| 문서 번호 | 문서명 | 비고 |
|----------|--------|------|
| SDPE-SAD-001 | 시스템 아키텍처 설계서 | 서브시스템 구조, CSC 정의 |
| SDPE-ICD-001 | 인터페이스 제어 문서 | 인터페이스 상세 명세 |
| SDPE-SRS-001 | 소프트웨어 요구사항 명세서 | 기능·비기능 요구사항 출처 |

## 2. 사용자 정의

### 2.1 사용자 역할

SDPE 운영자 콘솔에 접근 가능한 사용자 역할은 다음과 같다. 두 역할 모두 사내 운영팀 소속이며, 외부 사용자는 접근할 수 없다.

| 역할 | 설명 | 권한 범위 |
|------|------|----------|
| Administrator | 시스템 전체를 관리하는 최고 권한 사용자 | 전체 기능 접근 가능. 파이프라인 구성 변경, 처리 프로파일 관리, 감사 로그 조회, 사용자·계정 관리 포함 |
| Operator | 일상 운영을 담당하는 운영자 | 파이프라인 조회·실행, Job 관리·재처리, 모니터링, 알림 확인, 본인 계정 관리 |

### 2.2 사용자 책임

운영자 콘솔 사용자는 다음 업무를 수행한다.

- SAR 데이터 처리 파이프라인의 설계 및 구성
- 처리 작업(Job)의 실행·모니터링·재처리 판단
- 처리 큐 및 시스템 상태 감시
- 처리 실패·지연 등 이상 상황 감지 및 조치
- 처리 완료 제품의 품질 확인 및 관리
- 시스템 운영 이력(감사 로그) 관리

## 3. 유즈케이스 목록

### 3.1 파이프라인 관리

파이프라인은 SAR 위성 원시 데이터의 수신부터 최종 제품 생성까지의 처리 흐름을 정의하는 워크플로우(DAG)이다. 운영자는 위성·촬영 모드에 따라 파이프라인을 구성하고 관리한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 파이프라인 관리 | UC01 | 파이프라인 목록 조회 | 현재 운용 중인 파이프라인 목록을 조회한다. 위성, 촬영 모드, 처리 단계 수, 생성일 등 기본 정보를 확인할 수 있다. | ✓ | ✓ |
| | UC02 | 파이프라인 생성 | 새로운 처리 파이프라인을 생성한다. 파이프라인 이름, 대상 위성, 촬영 모드를 지정하고 데이터 입력 방식(수신 이벤트 기반 또는 파일 직접 입력)을 선택한다. | ✓ | ✗ |
| | UC03 | 파이프라인 수정 | 파이프라인의 기본 속성(이름, 대상 위성, 촬영 모드 등)을 변경한다. | ✓ | ✗ |
| | UC04 | 파이프라인 삭제 | 더 이상 사용하지 않는 파이프라인을 시스템에서 제거한다. | ✓ | ✗ |
| | UC05 | 파이프라인 복제 | 기존 파이프라인의 구성을 그대로 복사하여 새 파이프라인을 생성한다. 유사한 구성의 파이프라인을 빠르게 만들 때 사용한다. | ✓ | ✗ |
| | UC06 | 파이프라인 아카이브 | 당장 사용하지 않는 파이프라인을 아카이브 상태로 전환하여 운용 목록에서 분리한다. 아카이브된 파이프라인은 삭제되지 않으며 필요 시 복원할 수 있다. | ✓ | ✗ |
| | UC07 | 아카이브 파이프라인 조회 | 아카이브된 파이프라인 목록을 조회한다. | ✓ | ✓ |
| | UC08 | 파이프라인 복원 | 아카이브 상태의 파이프라인을 다시 운용 상태로 되돌린다. | ✓ | ✗ |
| | UC09 | 파이프라인 수동 실행 | 파이프라인을 운영자가 직접 실행하여 새로운 처리 작업(Job)을 생성한다. 자동 수신 이벤트가 아닌 운영자 판단에 의한 수동 기동이다. | ✓ | ✓ |

### 3.2 파이프라인 편집

운영자는 파이프라인의 처리 흐름을 DAG(방향 비순환 그래프) 형태의 캔버스에서 시각적으로 편집한다. 각 처리 단계를 노드로 표현하고, 노드 간 의존 관계를 엣지로 연결하여 데이터 처리 순서를 정의한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 파이프라인 편집 | UC10 | 처리 단계 추가 | 파이프라인에 새로운 처리 단계(원시 처리, 영상 처리, 카탈로그 등록 등)를 추가한다. | ✓ | ✗ |
| | UC11 | 처리 단계 삭제 | 불필요한 처리 단계를 파이프라인에서 제거한다. 데이터 입력 노드(수신 이벤트, 파일 입력)는 파이프라인의 시작점이므로 삭제할 수 없다. | ✓ | ✗ |
| | UC12 | 처리 단계 설정 | 각 처리 단계의 세부 설정을 편집한다. SAR 처리 단계에서는 수행할 작업을 선택하고, 작업 초기화 단계에서는 처리 프로파일과 우선순위를 지정하며, 파일 입력 단계에서는 입력 파일 경로를 설정한다. | ✓ | ✗ |
| | UC13 | 처리 단계 비활성화 | 특정 처리 단계를 일시적으로 비활성화하여 실행 시 해당 단계를 건너뛰도록 설정한다. 파이프라인 구성을 유지하면서 특정 단계만 제외할 때 사용한다. | ✓ | ✗ |
| | UC14 | 처리 흐름 연결 | 두 처리 단계 간에 새로운 의존 관계(엣지)를 설정한다. 선행 단계의 처리가 완료된 후 후행 단계가 실행되도록 순서를 정의한다. | ✓ | ✗ |
| | UC15 | 처리 흐름 연결 해제 | 두 처리 단계 간의 의존 관계를 제거한다. | ✓ | ✗ |
| | UC16 | 처리 단계 상세 조회 | 개별 처리 단계의 상세 설정 내용과 처리 결과 데이터를 확인한다. | ✓ | ✓ |

### 3.3 처리 프로파일 관리

처리 프로파일은 위성 종류와 촬영 모드에 따라 미리 정의된 처리 설정이다. 새 위성이 추가되면 해당 위성에 맞는 프로파일을 등록하는 것만으로 기존 파이프라인에 통합할 수 있다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 처리 프로파일 관리 | UC17 | 처리 프로파일 목록 조회 | 등록된 처리 프로파일 목록을 위성·촬영 모드별로 조회한다. | ✓ | ✓ |
| | UC18 | 처리 프로파일 생성 | 새로운 위성·촬영 모드 조합에 대한 처리 프로파일을 등록한다. 편파 구성, 처리 파라미터, 우선순위 등을 설정한다. | ✓ | ✗ |
| | UC19 | 처리 프로파일 수정 | 기존 처리 프로파일의 파라미터 설정을 변경한다. | ✓ | ✗ |
| | UC20 | 처리 프로파일 삭제 | 더 이상 사용하지 않는 처리 프로파일을 삭제한다. 해당 프로파일을 참조 중인 파이프라인이 있는 경우 사전 확인이 필요하다. | ✓ | ✗ |

### 3.4 처리 작업(Job) 관리

Job은 파이프라인의 1회 실행 단위이다. 위성 데이터 수신 시 자동으로 생성되거나, 운영자가 수동으로 생성할 수 있다. 운영자는 Job의 진행 상황을 추적하고, 실패 시 재처리를 지시한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| Job 관리 | UC21 | Job 목록 조회 | 전체 처리 작업 목록을 조회한다. 처리 상태(대기·실행 중·완료·실패)별로 필터링할 수 있다. | ✓ | ✓ |
| | UC22 | Job 상세 조회 | 개별 처리 작업의 실행 진행 상황을 확인한다. 각 처리 단계별 소요 시간, 현재 진행 단계, 오류 발생 내역 등을 상세히 확인할 수 있다. | ✓ | ✓ |
| | UC23 | Job 전체 재처리 | 실패하거나 완료된 처리 작업을 원시 데이터(Level-0)부터 다시 처리한다. 재시도 횟수가 초기화되며, 오조작 방지를 위해 Job ID 재입력 확인 절차가 적용된다. | ✓ | ✓ |
| | UC24 | Job 부분 재처리 | 특정 처리 레벨(Level-1, Level-2, Level-3)을 지정하여 해당 단계부터 재처리를 시작한다. 이전 단계의 결과물을 그대로 활용하므로 전체 재처리보다 효율적이다. | ✓ | ✓ |
| | UC25 | Job 취소 | 아직 처리가 시작되지 않은(대기 또는 할당 상태) 작업의 실행을 취소한다. | ✓ | ✓ |
| | UC26 | Job SLA 확인 | 처리 작업별 소요 시간이 목표 처리 시간(4시간) 이내인지 확인한다. 목표 시간 초과 시 지연 원인 파악의 기준이 된다. | ✓ | ✓ |

### 3.5 제품 관리

제품은 파이프라인 처리를 거쳐 생성된 SAR 영상 데이터이다. 운영자는 처리 완료된 제품의 메타데이터와 품질을 확인하고, 필요 시 재처리를 요청할 수 있다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 제품 관리 | UC27 | 제품 목록 조회 | 처리 완료된 SAR 제품 목록을 조회한다. 처리 레벨, 위성, 촬영 모드, 처리 상태 등으로 필터링할 수 있다. | ✓ | ✓ |
| | UC28 | 제품 상세 조회 | 개별 제품의 메타데이터를 확인한다. 촬영 영역(공간 범위), 촬영 시각, 해상도, 편파 구성, 처리 상태 등의 정보를 포함한다. | ✓ | ✓ |
| | UC29 | 제품 품질 확인 | 제품의 품질 검증 결과를 확인한다. 잡음 등가 시그마 영(NESZ), 피크 부엽비(PSLR), 기하 정확도, 방사 보정 품질 등 주요 품질 지표를 포함한다. | ✓ | ✓ |
| | UC30 | 제품 다운로드 | 제품 파일의 다운로드 URL을 발급받는다. 보안을 위해 발급된 URL에는 만료 시간이 설정된다. | ✓ | ✓ |
| | UC31 | 제품 미리보기 | 제품의 Quick-look 미리보기 이미지(Thumbnail)를 조회한다. 전체 파일을 다운로드하지 않고도 영상 내용을 빠르게 확인할 수 있다. | ✓ | ✓ |
| | UC32 | 제품 재처리 요청 | 특정 제품(촬영 단위)을 지정하여 원하는 처리 레벨부터 재처리를 요청한다. 품질 미달이나 파라미터 변경 등의 사유로 제품을 다시 생성할 때 사용한다. | ✓ | ✓ |

### 3.6 큐 모니터링

처리 큐는 파이프라인의 각 처리 단계에 작업을 분배하는 메시지 대기열이다. 운영자는 큐 상태를 감시하여 작업 적체, 처리 지연, 워커 장애 등을 조기에 감지한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 큐 모니터링 | UC33 | 큐 상태 조회 | 전체 처리 큐의 현재 상태를 조회한다. 큐별 대기 작업 수, 처리 워커(컨슈머) 수, 정상/비정상 상태를 확인할 수 있다. | ✓ | ✓ |
| | UC34 | 큐 추이 확인 | 최근 1시간 동안의 큐 대기 작업 수 변화 추이를 확인한다. 작업 적체 여부와 처리 속도 변화를 시각적으로 파악할 수 있다. | ✓ | ✓ |
| | UC35 | 대기 메시지 조회 | 큐에 대기 중인 개별 메시지를 우선순위별로 확인한다. | ✓ | ✓ |
| | UC36 | 실패 메시지 조회 | 최대 재시도 횟수(3회)를 초과하여 처리에 실패한 메시지(Dead Letter)를 확인한다. 장애 원인 분석의 진입점이 된다. | ✓ | ✓ |
| | UC37 | 처리량 확인 | 시간대별(1시간, 24시간) 처리 건수와 평균 처리 소요 시간을 확인한다. 시스템 전체의 처리 능력을 파악하는 데 사용한다. | ✓ | ✓ |

### 3.7 알림 관리

시스템은 처리 실패, 처리 지연, 품질 미달, 리소스 임계치 초과 등 이상 상황 발생 시 운영자에게 알림을 발행한다. 운영자는 알림을 확인하고 조치 완료를 기록한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 알림 관리 | UC38 | 알림 목록 조회 | 시스템이 발행한 알림 목록을 조회한다. 미확인 알림만 필터링하거나 전체 알림을 확인할 수 있다. | ✓ | ✓ |
| | UC39 | 알림 확인 처리 | 알림을 확인(Acknowledge)하여 운영자가 해당 상황을 인지하고 조치했음을 기록한다. 동시에 여러 운영자가 같은 알림을 처리하는 경우를 방지하기 위해 동시성 제어가 적용된다. | ✓ | ✓ |

> **알림 종류**
> - **재시도 초과**: 자동 재시도 3회가 모두 실패한 경우
> - **처리 지연**: 처리 작업이 목표 시간(4시간)을 초과한 경우
> - **품질 미달**: 제품 품질 검증 기준을 통과하지 못한 경우
> - **리소스 임계치**: 시스템 리소스(CPU, 디스크 등)가 임계값을 초과한 경우

### 3.8 운영 로그

운영자는 시스템의 동작 이력을 두 가지 관점에서 확인할 수 있다. 감사 로그는 누가 언제 어떤 조작을 했는지 기록하며, 실행 로그는 처리 과정의 기술적 상세를 기록한다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 감사 로그 | UC40 | 감사 로그 조회 | 시스템 운영 이력을 조회한다. Job 생성·할당·완료·실패, 파이프라인 실행, 재처리 요청, 알림 발행, 로그인 성공·실패, 사용자 생성·수정·역할 변경·비활성화, 비밀번호 초기화·변경 등 주요 이벤트의 발생 시각, 대상 Job 또는 사용자, 조작자 정보를 확인할 수 있다. 날짜 범위, Job ID, 조작자 등으로 필터링이 가능하다. | ✓ | ✗ |
| 실행 로그 | UC41 | 실행 로그 조회 | 처리 과정에서 발생한 기술적 로그를 조회한다. Job ID, 로그 수준(정보, 경고, 오류), 조회 건수 등으로 필터링하여 처리 오류의 원인을 분석할 수 있다. | ✓ | ✓ |

### 3.9 대시보드

대시보드는 시스템의 전반적인 운영 현황을 한눈에 파악할 수 있는 요약 화면이다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 대시보드 | UC42 | 운영 현황 조회 | 현재 처리 중인 작업 수, 최근 24시간 완료·실패 건수, 실패율, 평균 처리 소요 시간 등 시스템 운영 핵심 지표를 조회한다. | ✓ | ✓ |

### 3.10 인증

운영자 콘솔의 모든 기능은 인증된 사내 사용자에게만 노출된다. 인증은 JWT 기반이며, 역할(RBAC)에 따라 기능 접근이 제한된다. 모든 인증 이벤트는 감사 로그에 기록되어 보안 사고 분석의 근거로 사용된다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 인증 | UC43 | 로그인 | 사용자명과 비밀번호로 인증하고 접근 토큰을 발급받는다. 일정 횟수 이상 연속 실패 시 계정이 일시적으로 잠긴다. 성공·실패 모두 감사 로그에 기록된다. | ✓ | ✓ |
| | UC44 | 로그아웃 | 현재 세션의 토큰을 폐기하고 로그인 화면으로 복귀한다. | ✓ | ✓ |
| | UC45 | 토큰 갱신 | 접근 토큰 만료 시 갱신 토큰으로 자동 재발급한다. 갱신 실패 시 강제 로그아웃된다. | ✓ | ✓ |
| | UC46 | 본인 비밀번호 변경 | 로그인된 사용자가 현재 비밀번호를 확인한 뒤 새 비밀번호로 변경한다. 비밀번호 정책(최소 길이·복잡도)을 준수해야 한다. | ✓ | ✓ |

### 3.11 사용자 관리

Administrator는 운영자 콘솔에 접근 가능한 사용자 계정을 직접 등록·관리한다. 외부 가입 절차는 제공되지 않으며, 계정 생성은 전적으로 Administrator의 책임이다.

| 구분 | 유즈케이스 ID | 유즈케이스명 | 설명 | Admin | Operator |
|------|-------------|------------|------|-------|----------|
| 사용자 관리 | UC47 | 사용자 목록 조회 | 등록된 사용자 목록을 역할, 활성 상태, 검색어로 필터링하여 조회한다. 최근 로그인 시각도 함께 확인할 수 있다. | ✓ | ✗ |
| | UC48 | 사용자 생성 | 새 사용자 계정을 등록한다. 사용자명, 이메일, 역할(Administrator 또는 Operator), 초기 비밀번호를 지정한다. | ✓ | ✗ |
| | UC49 | 사용자 수정·비활성화 | 사용자의 역할, 활성 상태, 프로필 정보를 변경한다. 비활성화된 사용자는 이후 로그인이 거부되지만 과거 조작 이력은 감사 로그에 남는다. | ✓ | ✗ |
| | UC50 | 비밀번호 초기화 | 사용자 비밀번호를 임시 비밀번호로 초기화한다. 생성된 임시 비밀번호는 발급 시점 1회만 노출되며, 사용자는 최초 로그인 시 비밀번호를 반드시 변경해야 한다. | ✓ | ✗ |

## 4. 도메인 용어

| 용어 | 설명 |
|------|------|
| 파이프라인 | SAR 위성 데이터의 수신부터 최종 제품 생성까지의 처리 흐름을 정의하는 워크플로우. 방향 비순환 그래프(DAG) 형태로 구성된다 |
| 처리 작업 (Job) | 파이프라인을 1회 실행하는 단위. 위성 데이터 수신 시 자동 생성되거나 운영자가 수동으로 생성한다 |
| 처리 단계 (Step) | 파이프라인을 구성하는 개별 처리 과정. 원시 처리, 영상 처리, 카탈로그 등록 등이 있다 |
| 처리 프로파일 | 위성 종류·촬영 모드별로 미리 정의된 처리 설정. 새 위성 추가 시 프로파일 등록만으로 파이프라인 통합이 가능하다 |
| 제품 (Product) | 파이프라인 처리를 거쳐 생성된 SAR 영상 데이터. STAC 카탈로그에 등록되어 사용자에게 서비스된다 |
| 처리 큐 | 각 처리 단계에 작업을 분배하는 메시지 대기열. 큐 상태를 통해 시스템 부하와 처리 속도를 파악한다 |
| Dead Letter | 최대 재시도 횟수(3회)를 초과하여 처리에 실패한 메시지. 수동 조치가 필요한 장애 항목이다 |
| 아카이브 | 파이프라인을 삭제하지 않고 운용 목록에서 분리하는 상태. 필요 시 복원할 수 있다 |
| 접근 토큰 (Access Token) | 로그인 시 발급되는 단기 유효 토큰. API 요청 인증에 사용된다 |
| 갱신 토큰 (Refresh Token) | 접근 토큰 만료 시 재발급에 사용되는 장기 토큰. 유출 시 서버 측 폐기가 가능하다 |
| 역할 (Role) | 사용자에게 부여되는 권한 그룹. Administrator 또는 Operator로 구분된다 |

## 5. SAR 데이터 처리 레벨

SAR 위성 데이터는 다음 단계를 거쳐 순차적으로 처리된다. 각 단계의 결과물은 다음 단계의 입력으로 사용된다.

| 처리 레벨 | 처리 단계 | 설명 |
|----------|----------|------|
| Level-0 | 원시 데이터 처리 | 위성 전송 패킷 복원, 압축 해제, 레이더 신호 재구성 |
| Level-1 | SAR 영상 처리 | 레이더 신호를 영상으로 변환. SLC(복소 영상), GRD(강도 영상), 지형보정, 지도투영 제품 생성 |
| Level-2 | 영상 분석 | 생성된 영상에서 객체 탐지(선박, 차량 등), 변화 탐지, 지형 마스크 생성 등 분석 수행 |
| Level-3 | 응용 제품 생성 | 침수 지도, 산사태 탐지, 유류 오염 탐지, 작황 모니터링 등 특정 목적의 활용 제품 생성 |
