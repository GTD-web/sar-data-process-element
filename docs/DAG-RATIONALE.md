**문서 번호**: SDPE-DOC-DAG-001
**버전**: v1.0
**작성일**: 2026-04-20
**분류**: 내부 기술문서

## 1. 문서 목적

본 문서는 SDPE(SAR Data Processing Element) 운영자 콘솔의 파이프라인이 왜 단순 선형 리스트가 아닌 **DAG(Directed Acyclic Graph, 방향 비순환 그래프)** 로 모델링되어야 하는지, 그리고 왜 **그래프 캔버스 UI**가 필요한지를 정리한다. 또한 현재 시스템의 파이프라인 구성 현황과 향후 추가되어야 할 분기형 파이프라인 요구사항을 기술한다.

관련 문서:

- `docs/USECASE.md`
- `docs/USECASE-REPORT.md` (§3.1 파이프라인 관리, §3.2 파이프라인 편집)
- `SDPE-SAD-001` (시스템 아키텍처 설계서)
- `SDPE-ICD-001` (인터페이스 제어 문서)

## 2. DAG가 필요한 이유

### 2.1 SAR 데이터 처리의 본질이 "순서 있는 다단계 처리"이기 때문

SAR 위성 원시 데이터는 아래와 같은 다단계 가공을 거쳐야 최종 제품이 된다.

```
수신(EI-01) → Level-0(원시) → Level-1(영상) → Level-2 → 카탈로그 등록 → 배포
```

각 단계는 **이전 단계의 결과물을 입력**으로 사용하므로, 실행 순서와 데이터 전달 관계가 명시적으로 정의되어야 한다. 단순 리스트로는 "어떤 출력을 어떤 입력으로 쓰는가"를 표현하기 어려우며, 의존성을 그래프로 선언해야 스케줄러가 올바른 순서로 실행할 수 있다.

### 2.2 분기(fan-out)와 합류(fan-in)가 존재하기 때문

선형 파이프라인으로는 표현할 수 없는 구성이 실제 운영에서 필요하다.

- **편파(polarization)별 병렬 처리** — 한 장면에 대해 VV/VH 편파를 동시에 처리.
- **복수 타겟 레벨의 분기** — Level-1, Level-2, Level-3 제품을 각각 생성.
- **공통 후행 단계로의 합류** — 서로 다른 제품 레벨이 모두 카탈로그 등록·썸네일 생성으로 수렴.

이러한 분기/합류 구조는 DAG 없이 정확히 모델링·실행할 수 없다.

### 2.3 부분 재처리(UC24, UC32)를 지원하기 위해

`USECASE.md` UC24·UC32는 **특정 Level부터의 재처리**를 요구한다. 예를 들어 "scene_id와 target_level을 지정해서 Level-2부터 재실행"하려면, 노드 간 의존관계가 그래프로 저장돼 있어야 **해당 노드 downstream만 선택적으로 재실행**할 수 있다. 선형 스크립트라면 매번 처음부터 재실행해야 하며, 이는 CSC-08 재처리 요건을 충족할 수 없다.

### 2.4 위성·촬영 모드별로 처리 구성이 달라지기 때문

`USECASE-REPORT.md` §3.3에 정의된 **처리 프로파일**은 위성 종류와 촬영 모드별로 서로 다른 처리 흐름을 요구한다. 새 위성이 추가되었을 때 **파이프라인 DAG와 프로파일만 등록하면 기존 실행 엔진 재사용**이 가능해야 하므로, 파이프라인은 코드가 아닌 **데이터(그래프)** 로 취급되어야 한다.

### 2.5 순환 실행을 시스템이 막아야 하기 때문

CI 테스트 케이스(`cicd-workflow-diagram.html:219`)에 있는 *"DAG Builder > 순환 의존성을 감지하여 에러를 던진다"* 항목이 보여주듯, 엣지 추가 시 **사이클 검증**이 필수이다. 사이클이 허용되면 무한 루프·데드락이 발생할 수 있으므로 "방향 비순환"이라는 제약을 시스템이 강제해야 한다.

## 3. 왜 DAG 캔버스 UI가 필요했는가

### 3.1 운영자가 비개발자일 수 있다

`USECASE-REPORT.md` §2.1에 따르면 Administrator/Operator는 사내 운영팀 소속이며, 반드시 개발자는 아니다. 파이프라인을 **YAML/JSON으로 직접 편집**하게 하면 오타·들여쓰기 오류가 발생하기 쉽고, 전체 구조를 머릿속으로 그려야 하므로 인지 부담이 크다.

### 3.2 조작 단위 자체가 "노드와 엣지"이다

유즈케이스가 요구하는 편집 동작은 그래프 연산 그 자체이다.

| 유즈케이스 | 동작 |
|-----------|------|
| UC10 처리 단계 추가 | 노드 추가 |
| UC11 처리 단계 삭제 | 노드 삭제 |
| UC12 처리 단계 설정 | 노드 속성 편집 |
| UC13 처리 단계 비활성화(바이패스) | 노드 플래그 토글 |
| UC14 처리 흐름 연결 | 엣지 추가 |
| UC15 처리 흐름 연결 해제 | 엣지 제거 |

이 조작들을 **캔버스의 직접 조작(Direct Manipulation)** 으로 제공하면 텍스트 편집보다 오류 가능성이 현저히 낮아진다.

### 3.3 편집 모델과 모니터링 모델을 공유할 수 있다

Job 실행 중에는 같은 DAG 구조 위에 **노드별 진행 상태·소요시간·실패 위치**(UC22)를 표시할 수 있다. 편집 UI와 모니터링 UI가 **동일한 그래프 뷰 모델**을 공유하면 운영자의 학습 비용이 낮아지고, "어디에서 실패했는가"를 구조 맥락과 함께 즉시 이해할 수 있다.

## 4. 현재 시스템 상태와 공백

### 4.1 현황: 모든 mock 파이프라인이 선형 구조

`frontend/src/app/(planning)/_services/pipeline.mock.ts` 의 `toDAGSteps()` 함수는 입력된 steps 배열을 순서대로 `i → i+1` 엣지로만 연결한다.

```ts
// pipeline.mock.ts (발췌)
for (let i = 0; i < steps.length - 1; i++) {
  edges.push({ source: steps[i].order, target: steps[i + 1].order });
}
```

즉 DAG 스키마(`steps[] + edges[]`)는 갖추어져 있으나, **생성되는 모든 파이프라인이 분기·합류가 없는 단일 경로**이다. 이는 DAG 모델을 도입한 본래의 이유(§2.2, §2.3)를 실증하지 못한다.

### 4.2 공백: 분기형 파이프라인 샘플 부재

아래 시나리오를 보여줄 수 있는 mock/시드 데이터가 현재 없다.

1. **멀티 레벨 분기** — Level-1 생성 후 Level-2와 Level-3로 fan-out, 이후 공통 카탈로그 등록으로 fan-in.
2. **편파별 병렬 처리** — VV/VH 두 편파 처리 노드가 동시에 실행되고, 영상 합성 단계에서 fan-in.
3. **부분 재처리 진입점 분기** — TRIGGER 외에 재처리용 `LEVEL_ENTRY` 노드에서 중간 레벨로 진입.

이 샘플들이 없으면 DAG 에디터(UC10~UC16)의 기능 검증, 재처리(UC24/UC32)의 실행 시뮬레이션, 모니터링 화면의 분기 상태 표시 테스트가 모두 선형 케이스에 국한된다.

## 5. 권고 — 추가되어야 할 파이프라인 유형

아래 3개의 분기형 파이프라인을 mock 데이터 및 시드에 추가할 것을 권고한다.

### 5.1 다중 레벨 제품 생성 파이프라인 (fan-out → fan-in)

```
TRIGGER → L0 원시처리 → L1 영상처리 ┬─ L2 지형보정 ─┐
                                     ├─ L2 해양분석 ─┼─ 카탈로그 등록 → 썸네일
                                     └─ L3 분석준비 ─┘
```

- **목적**: 한 번의 수신으로 다수의 제품 레벨을 동시에 생성.
- **검증 포인트**: fan-out·fan-in, 부분 재처리(특정 L2만 재실행) 시 선택적 downstream 실행.

### 5.2 편파별 병렬 처리 파이프라인 (병렬 노드)

```
TRIGGER → L0 분리 ┬─ VV 처리 ─┐
                  └─ VH 처리 ─┴─ 영상 합성 → L1 → 카탈로그
```

- **목적**: Dual-pol 촬영 데이터의 병렬 처리.
- **검증 포인트**: 두 노드가 동시에 RUNNING 상태로 표시되는 모니터링, 한쪽 실패 시 합류 노드 상태 처리.

### 5.3 (삭제됨) 재처리 진입점 파이프라인

ICD가 "진입점별 **별도 파이프라인**"으로 설계를 확정했고 (`PARTIAL_L1_*` / `PARTIAL_L2_*` 가 이를 반영), 두 진입 노드를 한 DAG에 섞는 모델은 ICD 설계와 상충하여 제거되었다. UC32(제품 기반 재처리)는 기존 부분 재처리 파이프라인 방식으로 충분히 표현된다.

### 5.4 Quick-look 조기 분기 (side-branch)

```
TRIGGER → JOB_INIT → L0 → L1A ┬─ THUMBNAIL ──────────────┐
                              └─ L1B → L1C → L2 → L3 ────┴─ CATALOG
```

- **목적**: `L1A` 완료 시점에 Quick-look 미리보기를 바로 발행하여 운영자가 **L3까지 기다리지 않고** 장면 성공/실패를 조기 확인.
- **근거**: SAD §12.2 **CSU-07.06 Thumbnail Generator**, ICD `/v1/products/{id}/thumbnail` (UC31).
- **검증 포인트**: 메인 경로와 병행하는 side-branch, 서로 다른 깊이의 두 경로가 `CATALOG`에서 합류.
- **관련 유즈케이스**: UC31(제품 미리보기), REQ-FUNC-023(품질 조기 감지).

### 5.5 ICD/SAD와의 일치성

| 분기형 | SAD/ICD 근거 | 비고 |
|---|---|---|
| 5.1 다중 레벨 fan-out/fan-in | SAD §13.2 DAG 스케줄링 · §10.2 Level-2 CSU 다종 | 일치 |
| 5.2 편파 병렬 | ICD §1349 "편파별 별도 파일" · §1369 **TBD** (다채널 처리 방식) | **채널별 파일 TBD 가정 샘플.** 단일 파일로 확정 시 단일 L1A로 축약 필요 |
| ~~5.3 복수 진입점~~ | — | **삭제.** ICD는 진입점별 별도 파이프라인으로 설계 (`PARTIAL_L1_*` 등) |
| 5.4 Quick-look 조기 분기 | SAD §12.2 CSU-07.06 · ICD `/v1/products/{id}/thumbnail` | 일치 |

5.2는 DAG UI의 편파 병렬 수용성 검증을 위한 **탐색 샘플**이며, ICD §1369 TBD가 "단일 파일"로 확정되면 해당 mock은 제거/축약되어야 한다.

### 5.6 구현 스케치

`toDAGSteps()` 외에 명시적 엣지를 받는 빌더 헬퍼를 추가하고, 위 세 종류 각 1건 이상을 mock 목록에 포함한다.

```ts
// 예시: 분기형 빌더
function buildBranchedPipeline(
  steps: StepDef[],
  edges: { source: number; target: number }[],
): Pipeline { /* ... */ }
```

UI 측에서는 자동 배치(레이아웃) 알고리즘이 fan-out/fan-in을 시각적으로 구분할 수 있는지(예: dagre, elkjs) 사전 확인이 필요하다.

## 6. 요약

- DAG는 SAR 처리의 **다단계·분기·부분재처리** 요구를 표현하기 위한 필수 모델이다.
- 캔버스 UI는 **비개발자 운영자가 안전하게 그래프를 편집·관찰**하기 위해 도입되었다.
- 그러나 현재 모든 mock 파이프라인이 선형이어서 DAG의 도입 근거가 실증되지 않는다.
- **다중 레벨 fan-out/fan-in**, **편파별 병렬 처리**, **재처리 진입점** 세 유형의 분기형 파이프라인을 mock/시드에 추가해 DAG 모델의 실효성을 검증할 것을 권고한다.
